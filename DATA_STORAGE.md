# Хранение данных и следующие улучшения

## Как данные хранятся сейчас

Приложение хранит пользовательские данные локально на устройстве.

Основное хранилище: `IndexedDB`, база `personal-budget-private-v1`, объектное хранилище `state`, ключ `app`.

Резервная локальная копия: `localStorage`, ключ `personal-budget-private-v1-state`.

Серверная синхронизация, аккаунты и удалённая база сейчас не используются. Это удобно для приватности, но данные живут только в браузере/ PWA на конкретном устройстве.

## Как улучшить хранение

- Добавить ручной и автоматический экспорт резервной копии.
- Добавить шифрованный файл резервной копии с паролем.
- Добавить опциональную синхронизацию через личное облако пользователя.
- Добавить версионирование структуры данных и миграции между версиями.
- Добавить экран проверки резервной копии перед импортом.

## Курс валют

Требуемый сценарий нужно уточнить перед реализацией:

- ручной курс BYN/USD в настройках;
- история курсов по датам операций;
- автоматическая загрузка курса;
- пересчёт старых операций или фиксация курса на дату операции.

Без этого легко получить неверные расчёты накоплений и покупок.

## Главная страница

Текущий блок на главной частично дублирует нижнее меню. Хорошая замена: персональный финансовый дайджест.

Рекомендуемый подход:

- хранить каталог сообщений с условиями показа;
- показывать только сообщения, которые подходят текущему состоянию данных;
- хранить историю показов, чтобы сообщения редко повторялись;
- группировать сообщения по типам: накопления, платежи, категории, питомец, покупки, привычки.

Пример условия: сообщение `Вы почти накопили на ...` показывать, если до цели осталось меньше 20%.

Полная версия с 100+ сообщениями лучше делать отдельным этапом, потому что это уже отдельная подсистема правил и анти-повторов.

## Обязательные шаблоны

Раздел `Обязательное` сейчас фиксированный. Для шаблонов нужен отдельный слой данных:

- шаблон регулярного события;
- тип: расход, доход, пополнение накоплений, событие питомца;
- периодичность;
- дата начала;
- правило изменения: только текущий период, все периоды, только будущие.

Это лучше внедрять отдельной миграцией, чтобы не сломать существующие месяцы.
